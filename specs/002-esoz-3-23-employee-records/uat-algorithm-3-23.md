## Блок A — Реєстрація працівника (3.23.1.1–1.6)

**Мета:** Create Employee Request v2 з КЕП, прев’ю, запрошенням на email.

### A1. Доступ за роллю

1. Увійти як **OWNER** або **HR** або **ADMIN**.
2. Відкрити «Новий працівник» / створення запиту.
3. **Очікування:** форма відкривається; немає 403.  
   □ PASS / □ FAIL  

4. Повторити з **PHARMACY_OWNER** (якщо є pharmacy LE).  
   □ PASS / □ FAIL / □ N/A  

### A2. Статус запиту NEW (3.23.1.2.1)

1. На формі створення переконатися, що немає поля редагування `status`.
2. Зберегти чернетку (кнопка «Зберегти»).
3. У реєстрі заявок / реєстрі працівників чернетка має статус **«Новий»**.
4. **Очікування:** локальна чернетка = `NEW`, без uuid; доступні Edit / Delete.  
   □ PASS / □ FAIL  

### A3. Загальні відомості (3.23.1.2.2)

1. Обрати **тип працівника** з довідника (напр. DOCTOR).
2. Обрати **посаду** з довідника POSITION (або вільний текст для ADMIN/HR/RECEPTIONIST).
3. Вказати **дату початку роботи**.
4. Обрати **МНП** (division) з активних.
5. **Очікування:** усі поля зберігаються; кастомна посада лише для типів з `employee_type_custom_position_allowed`.  
   □ PASS / □ FAIL  

### A4. Персональні дані party (3.23.1.2.3)

1. Заповнити: прізвище*, ім’я*, по батькові (опц.), дата народження*, стать*, email*, телефони* (тип + номер).
2. **З РНОКПП:** `tax_id` = валідний ІПН; `no_tax_id` = false.
3. **Без РНОКПП:** `no_tax_id` = true; у `tax_id` — серія/номер документа; додати документ з `EMPLOYEE_IDENTITY_DOCUMENT_TYPES`.
4. Документ: тип*, номер*, дата видачі*; орган (опц.).
5. Для **PASSPORT**: окремо **серія** (2 кириличні літери) + **номер** (6 цифр), напр. АА + 123456.
6. Негатив: латинська серія `AA123456` → помилка формату UA.  
   □ PASS / □ FAIL  
7. Негатив: тип «Свідоцтво про народження» відсутній у списку / відхиляється.  
   □ PASS / □ FAIL  

### A5. Професійні дані для medical types (3.23.1.2.4)

Типи: DOCTOR, ASSISTANT, SPECIALIST, LABORANT, MED_COORDINATOR, MED_ADMIN, PHARMACIST.

1. Освіта*: країна, місто, заклад, номер диплома, ступінь, спеціальність.
2. Спеціальності*: ≥1 запис; **рівно одна** з `speciality_officio` = true.
3. Негатив: 0 основних → помилка «рівно одну».  
   □ PASS / □ FAIL  
4. Негатив: 2 основні → помилка «більше однієї».  
   □ PASS / □ FAIL  
5. Кваліфікації / науковий ступінь — за потреби (не обов’язкові всі поля, якщо не заповнені).

### A6. Прев’ю перед ЕП (3.23.1.4)

1. Натиснути «Завершити та підписати».
2. **Очікування:** спочатку модалка **«Перегляд запиту перед підписанням»** з ПІБ, типом, посадою, email, документами тощо.
3. Натиснути «Продовжити до підпису ЕП» → модалка КЕП.  
   □ PASS / □ FAIL  

### A7. Підпис КЕП і успіх (3.23.1.5–1.6.1)

1. Обрати КНЕДП, завантажити ключ, ввести пароль, підписати.
2. **Очікування успіху:** flash містить текст про успішне надсилання **і** що Система **автоматично надіслала запрошення на email** працівника.
3. Запит у списку: статус **«Новий»**, з’явився **id/uuid**, **немає** Edit/Delete (лише перегляд) — це pending eHealth, не чернетка.  
   □ PASS / □ FAIL  

### A8. Помилка API і повтор (3.23.1.6.2)

1. Спровокувати помилку (невалідні дані / sandbox error) або перевірити на вже надісланій чернетці після fail.
2. **Очікування:** повідомлення про помилку; можна виправити дані, знову прев’ю → КЕП → відправити.  
   □ PASS / □ FAIL / □ BLOCKED (якщо sandbox стабільний і fail не відтворити)

## Блок C — Перегляд запитів (3.23.2)

### C1. Перелік запитів

1. Відкрити реєстр заявок працівників.
2. Для кожного рядка перевірити наявність:
   - **ID запиту** (uuid або локальний id)
   - дата/час створення
   - ПІБ
   - статус (довідник EMPLOYEE_REQUEST_STATUS / локальні лейбли)
3. Фільтр за статусом (напр. NEW / APPROVED / REJECTED).  
   □ PASS / □ FAIL  

### C2. Деталі запиту

1. Відкрити View по запиту.
2. **Очікування:** id, status, inserted_at/created_at + поля з 3.23.1.2 (party, position, docs…).  
   □ PASS / □ FAIL  

### C3. Локальна чернетка vs pending

1. Чернетка без uuid, статус «Новий» → є Edit і Delete; текст delete згадує статус «Новий».
2. Після успішного Create v2 (є uuid, статус «Новий») → Edit/Delete **немає**.
3. Legacy SIGNED (якщо є) → лейбл **«Надіслано»**, не редагується як чернетка.  
   □ PASS / □ FAIL

## Блок D — Перелік і картка працівника + верифікація (3.23.3)

### D1. Фільтри реєстру працівників (3.23.3.1.1)

Перевірити фільтри:

1. Статус працівника  
2. МНП (division_id)  
3. **РНОКПП (tax_id)**  
4. Тип працівника (employee_type)  
5. **Загальний статус верифікації (verification_status)**  

□ PASS / □ FAIL  

### D2. Відображення в списку / на картці party

1. На картці party видно **РНОКПП** і **статус верифікації** (якщо заповнені).
2. У таблиці посад: посада, тип, МНП, статус (Активний / Новий / Звільнений…).
3. Пошук за ПІБ **незалежно від порядку слів** («Іван Петренко» = «Петренко Іван»).  
   □ PASS / □ FAIL  

### D3. Деталі працівника (3.23.3.2)

1. Відкрити Show для **DOCTOR** і для **SPECIALIST** (або ASSISTANT).
2. **Очікування:** блоки освіта / спеціальності / кваліфікації / науковий ступінь видимі для **всіх** medical types, не лише DOCTOR.  
   □ PASS / □ FAIL  

### D4. Попередження NOT_VERIFIED (3.23.3.2.2)

Відкрити верифікацію party з `NOT_VERIFIED` за напрямками. Тексти мають відповідати ТЗ:

**DRFO:** РНОКПП / ДН / ПІБ не відповідають ДПС; при відмові від РНОКПП — оновити документ.  
□ PASS / □ FAIL / □ N/A  

**DRACS death:** актовий запис про смерть потребує підтвердження/спростування.  
□ PASS / □ FAIL / □ N/A  

**DMS passport:** паспорт недійсний за даними ДМС; звірити з оригіналом.  
□ PASS / □ FAIL / □ N/A  

Загальний абзац про обмеження роботи в ЕСОЗ / звернення до НСЗУ / ДПС / ДРАЦСГ / ДМС.  
□ PASS / □ FAIL  

### D5. Підтвердження / спростування смерті (3.23.3.4)

1. Для `dracs_death` = NOT_VERIFIED відкрити оновлення статусу.
2. **Підтвердити смерть:** VERIFIED + reason **MANUAL_CONFIRMED** + коментар.
3. **Спростувати:** VERIFIED + reason **MANUAL_NOT_CONFIRMED** + коментар.
4. Негатив: без коментаря (якщо UI вимагає) — не відправляє.  
   □ PASS / □ FAIL / □ N/A  

### D6. Дії після перегляду (3.23.3.3)

1. З картки верифікації / працівника доступне **оновлення** даних (п. 3.23.1).
2. Доступна **деактивація** активного (п. 3.23.4).  
   □ PASS / □ FAIL

## Блок E — Деактивація (3.23.4)

### E1. Доступ

1. Під OWNER / HR / ADMIN / PHARMACY_OWNER на APPROVED працівнику є дія «Звільнити» / «Деактивувати».
2. Модалка відкривається з заголовком про деактивацію.  
   □ PASS / □ FAIL  

### E2. STOPPED

1. Обрати статус **Припинено (STOPPED)**.
2. Вказати `end_date`: ≥ `start_date`, ≤ сьогодні.
3. Негатив: end_date < start_date → помилка.
4. Негатив: end_date > today → помилка.
5. Підтвердити; працівник стає неактивним / STOPPED.  
   □ PASS / □ FAIL  

### E3. ENTERED_IN_ERROR

1. На іншому APPROVED: статус **Внесений помилково**.
2. Поле end_date **порожнє / не передається**.
3. Успішна деактивація.  
   □ PASS / □ FAIL  

### E4. Тексти попередження

1. Для **DOCTOR:** текст згадує автоматичне розірвання декларацій.
2. Для **інших типів:** текст без блоку про декларації, але про незворотність і деактивацію ролей.  
   □ PASS / □ FAIL
